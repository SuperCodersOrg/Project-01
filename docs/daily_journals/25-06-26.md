Daily Design Journal · Day 3
Section 1 – Specific Bug

The biggest issue today happened while implementing functors for my HashMap. I got a huge wall of compiler errors because my DefaultHash template was declared after the HashMap template.

error: 'DefaultHash' does not name a type
error: 'DefaultHash' is not a class template
error: 'HashMap' is not a template

It looked like hundreds of errors but most of them were actually caused by that single mistake.

Later while converting everything into .h and .tpp files I also got another compiler error.

error: expected initializer before '<' token

That happened because I included my .tpp file before all the declarations inside the header.

Section 2 – Failed Attempt

The first thing I did after seeing all those template errors was start fixing them one by one. That honestly wasn't helping because every error depended on the previous one.

After reading through the file carefully I realised the declaration order itself was wrong. HashMap was trying to use DefaultHash before the compiler even knew it existed. Once I moved the declarations around, almost all the remaining errors disappeared.

I also wanted to implement operator[] exactly like STL's unordered_map. I got most of it working but once rehashing happened inside the operator, things started breaking. The load factor wasn't getting updated correctly and inserting new keys during operator[] caused more bugs than expected. Instead of forcing it, I removed the operator for now and decided to come back to it later after the rest of the HashMap was stable.

While moving everything into .h and .tpp files I also tried using AI to split the code automatically, but it kept changing parts of my implementation or formatting things differently. In the end it was faster to copy everything manually and test each file afterwards.

Section 5 – Learning Reflection

Today I understood template compilation much better than before.

Before this I knew templates had to be in header files, but I never really understood why declaration order mattered so much. The compiler only knows what it has already seen, so even a correctly written template can produce dozens of confusing errors if something is declared later than it's supposed to be.

I also learnt that sometimes removing a feature temporarily is better than forcing it into the project. I spent some time trying to make operator[] work exactly like STL, but it started interacting with rehashing and load factor updates in ways I hadn't planned for. Instead of making the rest of the HashMap unstable, I decided to leave it out for now and come back to it once everything else is fully tested.

Converting the whole project into .h and .tpp files also made me realise that changing project structure is different from writing new code. Even if the implementation stays the same, it's really easy to introduce include-order bugs or template compilation issues if you're not careful.

By the end of the day the HashMap was working with custom key types through functors, the project had been reorganized into proper header and implementation files, everything was building correctly with CMake, and I had started the Redis Lite project on top of the data structures I built. I also implemented the basic CLI along with the SET and GET commands, so the next step is expanding the command set and testing it more thoroughly.
